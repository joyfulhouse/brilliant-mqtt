# Voice satellite guide

A Brilliant Control panel can act as a **Home Assistant ESPHome voice
satellite**: always-on wake word on the panel's built-in mic, speaker playback
of TTS responses. STT, the conversation agent, and TTS all run in your HA
Assist pipeline — the panel is fully backend-agnostic (works with whatever STT /
LLM / TTS you already run). The integration handles install, updates, and repair
automatically.

## Requirements

- An HA **Assist pipeline** configured with STT, a conversation agent, and TTS
  (any combination you already run — Local, OpenAI, Whisper, etc.).
- The panel must be reachable from the HA host (network path for the ESPHome
  satellite API, port 6053).
- The HA host must be able to reach **GitHub** once per release to download the
  ~57 MB voice payload asset. It is cached after the first panel install and
  reused across the fleet.

## Enable voice

**Option A — during onboarding (recommended):**

On the **Panel settings** step of the Add Integration flow, toggle **Enable
voice satellite**, choose a **Wake word** (`okay_nabu`, `hey_jarvis`, or
`hey_mycroft`), and optionally set a **Home Assistant host override** (see
[Configuration](#configuration) below). The integration installs the satellite
alongside the bridge agent in the same SSH session.

**Option B — after onboarding:**

On the panel's device page in HA, flip the **Voice satellite** switch to `on`.
The integration downloads (if needed) and installs the satellite over SSH, then
starts it. Use the **Wake word** select to choose or change the wake word at
any time (the satellite restarts automatically).

## Connect to Home Assistant

Once the satellite is running, HA discovers it automatically:

1. HA sees a `_esphomelib._tcp` zeroconf advertisement from the panel.
2. A **new device discovered** notification appears — accept it to add the
   ESPHome device. The brilliant_mqtt integration is MQTT-only; HA's built-in
   **ESPHome** integration owns this device entry.
3. Go to **Settings → Voice assistants**, find the new `assist_satellite`
   entity for the panel, and assign your Assist pipeline.
4. Talk to the panel — say your wake word and ask something.

**If discovery doesn't appear:**

- Confirm the satellite is running: check `brilliant-voice` under the panel's
  HA device page (bridge health should be `ok`).
- Confirm port 6053 is reachable from the HA host. The panel's nftables ruleset
  is updated at every satellite start; if the accept rule is missing, pressing
  the **Repair bridge** button redeploys and reapplies it.
- If the panel is on a segmented IoT VLAN, set `VOICE_HA_HOST` (see below) so
  the panel can reach HA's TTS endpoint, and confirm your VLAN/firewall lets the
  HA host reach the panel on port 6053 (HA connects in to the satellite's API).

## Wake words

Three models are bundled (microWakeWord, runs on the panel's ARMv7 A9 CPU):

| Wake word | Phrase |
|---|---|
| `okay_nabu` (default) | "Okay, Nabu" |
| `hey_jarvis` | "Hey, Jarvis" |
| `hey_mycroft` | "Hey, Mycroft" |

Change the active wake word via the **Wake word** select entity on the device
page — no reinstall needed, the satellite restarts with the new model.

## Configuration

The integration writes `/etc/brilliant-voice.env` on the panel (mode `0600`).
You do not edit this file directly; it is managed by the integration and
regenerated on every repair/update.

| Variable | Default | Notes |
|---|---|---|
| `BRILLIANT_PANEL` | _(set from entry)_ | Required. Panel slug used for MQTT topics. |
| `VOICE_NAME` | `Brilliant <panel>` | Satellite display name in HA. |
| `VOICE_API_PORT` | `6053` | ESPHome native API port. HA connects inbound on this port. |
| `VOICE_WAKE_WORD` | `okay_nabu` | Active wake word model. Managed via the Wake word select entity. |
| `VOICE_HA_HOST` | _(empty)_ | `hostname=ip` mapping (e.g. `homeassistant.example=10.0.0.5`). Only needed when the panel can't resolve your HA URL's hostname via DNS — common on segmented IoT VLANs. Leave blank if the panel's DNS resolves HA's hostname normally. |
| `VOICE_ENABLE_AEC` | `off` | Enable acoustic echo cancellation (AEC). Off by default — the satellite keeps the mic closed during TTS playback, so normal use has no echo problem. Enable only for barge-in support. |
| `VOICE_MIC_DEVICE` | `default` | ALSA capture device. The panel's `default` device is Brilliant's tuned DSP chain (DC removal + 30× gain + mono downmix), which gives far-field "hey jarvis" scores ~0.996. Do not change unless you know what you're doing. |
| `VOICE_SND_DEVICE` | `plug:dmix_48000` | ALSA playback device. Mixes TTS output with other panel audio. |

Advanced AEC variables (`VOICE_AEC_MIC_DEVICE`, `VOICE_AEC_DELAY_MS`,
`VOICE_AEC_TYPE`) and `LOG_LEVEL` are documented in
`src/brilliant_voice/config.py`.

## Troubleshooting

**Satellite not discovered by HA:**
- Check the **Voice satellite** switch is on and the bridge health is `ok`.
- Confirm the panel can reach the HA host on port 6053 (inbound) and that HA
  can reach the panel on that port.
- Press **Repair bridge** — it reapplies the nftables accept rule.
- If on a segmented VLAN, set `VOICE_HA_HOST` (see above).

**Wake word not detected:**
- Say the phrase slowly and clearly within 2–3 metres of the panel.
- Check `LOG_LEVEL=DEBUG` output for wake word scores (SSH in and `journalctl
  -u brilliant-voice -f`).
- The default `VOICE_MIC_DEVICE=default` uses the panel's tuned DSP chain —
  do not change it to the raw dsnoop tap (scores drop significantly at distance).

**No TTS audio after wake + response:**
- The panel needs to reach HA's TTS audio URL. If HA's hostname is
  unresolvable from the panel, set `VOICE_HA_HOST=ha.example=10.0.0.5` (the
  `hostname=ip` format is written to `/etc/hosts` on the panel at every start).
- Confirm `VOICE_SND_DEVICE` is `plug:dmix_48000` (the default).

**`voice_missing` repair issue raised in HA:**
Voice is enabled for this panel but the satellite is not running. This happens
if the `/var/brilliant-voice/` payload was lost (unusual). Press **Repair** on
the issue card (or the **Repair bridge** button) — the integration reinstalls
the satellite from the cached or freshly downloaded payload. The release asset
must exist for the installed integration release version.

## Advanced

**AEC (acoustic echo cancellation):** ships off. Enable via `VOICE_ENABLE_AEC=1`
(or toggle in a future integration UI option). Only useful for barge-in (talking
while TTS plays). When AEC is on, a sidecar daemon reads the raw two-mic dsnoop
tap and feeds cleaned audio to LVA; it uses the panel's own `audio_dsp` Cython
module (panel py3.10 only).

**Resource caps:** the `brilliant-voice` systemd unit is capped at `Nice=5`,
`MemoryMax=300M`, `CPUQuota=100%`, `OOMScoreAdjust=500`. Always-on wake
inference runs at approximately 12× realtime on the panel's ARMv7 A9 CPU and
cannot starve the touchscreen UI.

**Coexists with Alexa:** the mic is shared via ALSA dsnoop, so the satellite
and the panel's built-in Alexa vassal (Pryon Lite uwsgi) both capture from the
same hardware mic simultaneously. No Alexa disable is required or performed.

**Building the payload yourself:** see [deploy/voice/README.md](../deploy/voice/README.md).
The script produces the ~57 MB tarball; after building, upload it as a GitHub
release asset so the integration can fetch it.
