## Interpretation

### Direct device capabilities

Types such as LOCK, DOORBELL, MUSIC, THERMOSTAT, MOTION_SENSOR, LIGHT, OUTLET, GENERIC_ON_OFF, ALWAYS_ON, SHADE, CAMERA, GARAGE_DOOR, WEATHER, CLIMATE_SENSOR, BRILLIANT_PLUG, SECURITY_SYSTEM, WATER_SHUTOFF_VALVE, SOLAR, and HOME_ENERGY_SYSTEM describe end-user capabilities.

### Configuration and orchestration

Types ending in `_CONFIGURATION`, plus EXECUTION, SCENE_EXECUTION, STATE_CONFIGURATION, DEVICE_GROUPS_CONFIGURATION, CONFIGURATION_TEMPLATES, HOME_DETAILS, PROPERTY, BOOTSTRAP, REQUEST_DISPATCHER, and OAUTH_CONFIGURATION describe catalogs, lifecycle, credentials, bindings, automation, or provisioning rather than an ordinary HA entity.

### Infrastructure

CONTROL, DISCOVERY, STATUS, REMOTE_BRIDGE, OBJECT_STORE, AV_STREAMER, UI, VOICE, HARDWARE, WIFI, ANALYTICS, UART status types, REMOTE_MEDIA, BLE, NOTIFICATION, DEVICE_METRICS, THIRDPARTY_DISCOVERY, and USB_ETHERNET are panel/platform services or diagnostic surfaces.

### Partner adapter evidence

The many partner-specific configuration types demonstrate the breadth of the native peripheral-hosting framework. They do not imply that a matching local device exists or that the adapter is cloud-independent.

## Versioning rule

On firmware update, diff both directions:

- old value with changed name;
- new name/value;
- removed interface package;
- changed required fields;
- live instance type mismatch.

Unknown future values must map to `UNKNOWN` behavior rather than being treated as an adjacent known type.
